## Need 1 — Folding residuals off a clean spine

**1. Backward-slice gathering — Head et al., CHI 2019 ("code gathering tools" for Jupyter).** Select a result; the tool slices backward through the execution log to gather the *minimal ordered code* that produced it, leaving the mess behind — and archives past result versions with the slices that produced them. Transplant: this is the reducer's job stated as an algorithm — the spine is the gathered slice from the session's outcomes; every turn not on any outcome's slice is residual by definition. Breaks: notebooks have a static dataflow graph to slice over; sessions don't — slice membership is an LLM judgment, so residual/spine classification errors are a first-class failure mode, not an edge case.

**2. Curate-by-hide with lossless export — incident.io timeline curation.** Everything in the incident channel is auto-captured; humans pin key events (pin lands at the event's *original* timestamp, not pin time) and hide irrelevant items; the postmortem export includes only unhidden items; hidden ≠ deleted. Transplant: reducer auto-curates, but Pi gets the same gestures — demote-to-residual and promote-to-spine — and his overrides are curation state layered on the map, with export/print honoring them. Breaks: incident.io curation is post-hoc and manual; cartographer's curator is a live LLM — Pi's overrides must be durable pins the reducer cannot silently overwrite on its next pass, or human curation evaporates every update cycle.

**3. Three-state visibility cycling — org-mode TAB / Shift-TAB.** Per-heading TAB cycles folded → children-only → full subtree; global Shift-TAB cycles overview → contents → show-all. Two keys give total density control with zero mouse targets. Transplant: per-node disclosure states (title-only → node face + residual stubs → residuals expanded) plus a global overview/contents/all cycle for the whole map. Breaks: org fold state is ephemeral per-visit; cartographer's folds carry *semantics* (residual = folded-by-default permanently) — the default fold state must be data, with view-state layered separately on top.

**4. Resolved-thread collapse to a stub — GitHub resolved conversations.** A resolved thread collapses to a single summary line with "Show resolved"; content fully preserved, visually retired, reversibly. Transplant: a dead-end detour folds to a one-line stub under its parent node — "false alarm: 'missing code repo' — no code ever existed (6 turns)" — with the whole wrong turn behind a click. Breaks: GitHub's stub line is the thread's first comment (verbatim); cartographer's stub is *generated* — the stub summary itself can misrepresent the detour, so the stub needs the same quote-anchored validation as any node.

**5. Frame merging with a count badge — flame graphs (Brendan Gregg convention; DevTools/Perfetto).** Identical adjacent stack frames merge into one wide frame; repetition renders as one element with aggregate weight instead of N copies. Transplant: retry churn (edit → run tests → fail → edit ×5) merges into one node "fix iteration ×5" with the individual attempts as residuals behind it. Breaks: flame-graph merging is exact string equality; session retries are only *semantically* similar — merging can bury the one iteration that changed the approach, so the merge boundary needs the validate pass too.

**6. Subgoal-boundary compression over a retained raw layer — MAGE (arXiv 2606.06090).** Two-layer tree: bottom layer keeps every raw action-observation pair; top layer compresses contiguous completed segments into summary nodes at subgoal boundaries; summaries carry pointers to exactly the raw nodes they cover. Transplant: this *is* the map schema shape — top layer = map nodes, bottom layer = transcript spans, "covered nodes" = provenance pointers; folding is only legal at subgoal-completion seams. Breaks: MAGE compresses for the agent's context budget; cartographer folds for human reading — the fold criterion is "does Pi need this visible to understand the session," which no token threshold approximates.

**7. Structurally lossless trimming floor — CMV (arXiv 2602.22402).** Three-pass trim preserves every user message, assistant response, and tool *invocation* verbatim while unconditionally stripping mechanical bloat: raw tool outputs over a threshold (stubbed with an indicator), base64 images, thinking blocks, usage metadata. Transplant: a principled default for what residual folding may elide from view (mechanical payloads) versus what must always remain reachable (intent-carrying turns) — the trim rules double as the residual taxonomy. Breaks: CMV trims for *replay into a model*; for the human, the stripped tool output is sometimes the whole point (the log line that was the root-cause evidence) — cartographer must stub-with-pointer, never stub-and-discard.

**8. Output collapse with fixed-height scroll — Jupyter scrolled outputs.** Long cell outputs collapse to a fixed-height scrollable region or a collapsed indicator, so one verbose output can't destroy page readability. Transplant: any raw payload surfaced inline in the map (a quoted log, a diff) renders inside a capped, scrollable well — the spine's vertical rhythm survives arbitrary payload size. Breaks: little; the trap is using it so much the map becomes a wall of peepholes — capped wells are for residual drill-down, not spine content.

## Need 2 — Node history / updated-in-place affordances

**9. Page-plus-history-tab — MediaWiki/Wikipedia.** The article shows only current state; the complete revision list lives behind one tab, with any-two-revision diffs and per-revision attribution/comment. The reader never encounters history unless they ask. Transplant: node face = current version only; a small version chip (v3) opens the revision list with diffs — "reads clean top-to-bottom" is exactly the article/history separation. Breaks: wiki revisions have human authors and edit summaries; node versions are reducer rewrites — the "why" of each version must be the triggering transcript span, or history is a list of unexplained mutations.

**10. Checkpoint auto-collapse + named versions — Figma version history (also Google Docs version grouping).** Autosave checkpoints accumulate every ~30 min and on crash/disconnect, but stay visually minor; *named* versions (title + description) are deliberate, permanent landmarks; restore never destroys — it checkpoints current state first. Transplant: reducer micro-updates are autosave-grade (collapsed, grouped); only meaning-changing updates mint a visible version bump v1→v2 with a one-line "what changed" description. Restore/rollback of a node is itself a new version. Breaks: Figma's named/auto distinction comes from a human's intent; cartographer must *classify* semantic vs cosmetic updates automatically — misclassifying spams version chips (trust erosion) or hides a real reversal (worse).

**11. "Changed since you last looked" — Google Docs "See new changes" / GitHub viewed-checkbox semantics.** Docs diffs the document against your last visit; GitHub's per-file "Viewed" checkbox collapses a reviewed file and *auto-unchecks itself when the file changes*. Transplant: the map stores Pi's last-seen version per node; nodes updated since then carry a glint, and a "what's new since my last visit" lens shows only those diffs — the single most valuable affordance for a map Pi consults intermittently. Breaks: live viewing blurs "visit" boundaries — needs an explicit mark-as-caught-up action rather than an inferred one; and per-viewer seen-state must live outside the map data.

**12. Ignore-revs history filtering — `git blame --ignore-revs-file`.** Mechanical commits (reformatting, renames) are registered as ignorable so blame/history skips them and shows the last *meaningful* change. Transplant: node history renders meaning-bearing versions by default; reducer housekeeping versions (re-phrasings with no semantic delta) are filterable out of the history list. Breaks: git's ignore list is human-curated per commit; cartographer's is an LLM judgment — a wrongly ignored version hides a real change, so demote-behind-a-toggle is safe where drop is not.

**13. Stable identity + any-pair version diff — Gerrit patchsets.** A change is one stable identity; each push is a numbered patchset; the UI diffs *any two* patchsets, not just adjacent ones, and comments attach to specific patchsets. Transplant: node identity is stable and URL-addressable; versions are ordinals; the history view supports diff(v1, v4) directly — essential for "what changed since my annotation" when several bumps happened underneath. Breaks: patchset diffs are text diffs of one file set; node versions mutate structured fields (claims, status, quotes) — needs a field-level semantic diff renderer, not a unified text diff.

**14. Validate-before-trust — MAGE's Maintain operation (arXiv 2606.06090).** Immediately after each compression, an LLM validates the summary against the exact raw nodes it covers ("accurately captures the subgoal, nothing critical omitted?"); on failure, the diagnostic is recorded on the node and Revise rolls the state back to the boundary and branches — the bad summary never enters trusted state. Ablation: removing Maintain costs 5.2–6.7 points of task success. Transplant: every node create/update is checked against its provenance spans before it renders as trusted; a failed check renders visibly quarantined (flagged, not hidden, not trusted). The brief's trust-extinction concern gets a concrete mechanism and an empirical justification. Breaks: it's an extra LLM call per update (budget pressure), and a validator can share the summarizer's blind spots — quote-anchored checks (does the verbatim quote exist in the span?) should backstop the LLM judgment.

**15. Named-snapshot lineage — CMV (arXiv 2602.22402).** Session states are immutable named snapshots with metadata (name, timestamp, source session, tokens, tags) and parent pointers forming lineage chains; `Tree(G)` renders "git log --graph for conversational context"; branch spawns a new session from any snapshot with an orientation message. Transplant: map-level (not node-level) versioning — snapshot the whole map at meaningful seams (pre-compaction, phase gates, session fork) with names and lineage; if Pi ever forks a session, the map forks with matching lineage. Breaks: CMV's DAG exists to *reuse context across sessions*; cartographer v0 is single-session/post-hoc — map lineage is schema-level insurance, not a v0 surface, and building its UI early is scope creep.

## Need 3 — Provenance drill-down to source spans

**16. Detail-on-select pane — Chrome DevTools Performance / Perfetto / Jaeger.** Selecting a span populates a persistent detail pane (args, duration, links) *without navigating away*; the overview never scrolls out from under you; selection is the sole driver of the pane. Transplant: clicking a node/edge/claim fills a side pane with current state, version chip, provenance pointers, and the verbatim quotes — the map itself never leaves the screen. Breaks: trace viewers select exactly one span; Phase 2 requires *multi-select with payloads* — the pane must gracefully become a selection-set inspector, which no trace viewer models.

**17. Span-to-log deep link — Jaeger/OpenTelemetry trace-log correlation.** A span carries pointers to the exact log records within its bounds; one click opens logs pre-filtered and pre-scrolled to the span's window. Transplant: provenance pointer = transcript path + span; click opens a transcript viewer scrolled to and highlighting exactly those lines, with the map still visible — the "prove it" gesture in one hop. Breaks: log stores are append-only with stable IDs; transcript JSONLs get rewritten at compaction — raw line numbers are not durable (see 18).

**18. Multi-selector redundant anchoring — W3C Web Annotation Data Model + Hypothesis fuzzy anchoring.** An anchor stores several redundant selectors (TextQuoteSelector with prefix/suffix, TextPositionSelector, structural path); resolution cascades exact → position → fuzzy quote, so any surviving locator re-anchors it. Transplant: a provenance pointer is a bundle — {transcript path, message UUID, line range, exact quote + hash} — so compaction-rewritten or re-serialized transcripts still resolve via UUID or quote. Breaks: fuzzy quote-matching was built for third-party pages drifting slightly; the failure stats in 27 say don't lean on fuzz — lean on message UUIDs (Claude Code transcripts have them) and keep the quote as the audit artifact.

**19. Quote survives the anchor — Hypothesis highlights / Google Docs comments on deleted text.** The annotation stores the selected text itself, so even a fully orphaned annotation still *shows what it referred to* (Docs displays the quoted text with the comment after the target is deleted). Transplant: decision nodes carry their verbatim transcript quote *in the node*, visually distinct (quotation styling) from reducer-inferred prose — if the span pointer ever dies, the quote is self-sufficient evidence; the renderer's quoted-vs-inferred distinction falls out of storage format. Breaks: little — the discipline required is that quotes are never elided/ellipsized by the renderer, because truncation quietly re-introduces paraphrase.

**20. Execution counter vs document order — Jupyter `execution_count`.** Every executed cell is stamped with a monotonic counter, exposing true execution order against document order — the two orders visibly disagree. Transplant: every node badge-carries its turn-range, so reading order (spine) and chronological order (transcript) are both always recoverable; a "chronology" lens can re-sort or number nodes by time when Pi suspects the clean narrative is hiding sequence-dependence. Breaks: Jupyter's counters are famous for *confusing* users — chronology must stay an on-demand lens, never the primary axis, or cartographer regresses to the log-faithful tools it exists to replace.

**21. Narrative order decoupled from execution order — Observable reactive notebooks.** Cells execute in dataflow (topological) order regardless of document position, so authors arrange the document for *reading* while the runtime maintains correctness — the document is narrative, the graph is truth, and the system owns the mapping. Transplant: the philosophical license for the whole product: the spine is arranged for comprehension, the transcript holds the truth, and the reducer owns the mapping between them. Breaks: Observable's mapping is *provably* correct (static dependency analysis); the reducer's is probabilistic — cartographer must earn the reader's trust per-node (validate pass, provenance one hop away) rather than inherit it from the architecture.

**22. Line-range history — `git log -L` / `--follow`.** History of a specific line range, followed across file renames and moves — "how did this exact statement get here" as a first-class query. Transplant: claim-level provenance — select an assumption *inside* a node and trace it back through node versions to the turn where it first entered the map. Breaks: `-L` works by textual heuristics; claims get reworded across reducer versions — this only works if claims are schema-level objects with stable IDs inside nodes, which is a real schema cost decided in Phase 3.

## Need 4 — Annotation on mutating targets, review batching, lifecycle closure

**23. Pending-review batch — GitHub PR reviews.** Inline comments accumulate as "Pending," visible only to the author, attached to precise locations; one Submit publishes them as a single review with a summary verdict — one notification, one unit of instruction. Transplant: the brief's review packet verbatim: selections + comments accumulate client-side as pending, submit once, land as one coherent instruction set. Breaks: a GitHub review targets a frozen artifact (the diff at a head SHA); a live map can mutate *while Pi composes* — submit needs a staleness check ("node X changed while composing — review your comment") that GitHub never needed.

**24. Outdated-comment mechanic — GitHub PR review comments (docs + discussions #23138).** Each comment anchors to a line position in the diff *of a specific commit*; when later pushes change that region, the comment flips to "outdated," collapses out of Files Changed, but persists with its original diff-hunk context attached — you can always see what it referred to, and the Conversation tab keeps the thread alive. Transplant (the brief's chosen transplant — mechanics confirmed): annotation records node@version at creation; node bump ⇒ outdated badge + "view what changed since" (diff v_annotated → v_current); the annotation keeps its as-annotated payload the way GitHub keeps the hunk. Breaks: GitHub outdates on *any* touching push — including whitespace and, for file-level comments, any push at all (community discussion #86527: a top complaint) — so cartographer must outdate only on semantic change to the annotated *element*, or annotation churn under an active reducer destroys the feature.

**25. Ported comments — Gerrit 3.5.** Unresolved comments from older patchsets are automatically re-positioned onto the latest patchset's diff (endpoints `ported_comments/`), so open items can't be missed by looking only at the newest version; resolved ones stay put in history. Transplant: *unresolved* annotations follow the node forward — re-anchored onto the current node face when mapping succeeds — instead of being buried under "outdated"; resolution state, not age, decides visibility. Breaks: Gerrit's textual position-mapping mis-places comments after heavy rewrites, and users report old-comment clutter (repo-discuss thread) — when semantic mapping fails, fall back to outdated-with-diff (24) rather than mis-anchor; and port only unresolved, or the map silts up.

**26. Unresolved-comments-block-submit — Google Critique (Sadowski et al., ICSE-SEIP 2018).** Comments carry a resolved/unresolved bit; unresolved = action item by definition; a change needs zero unresolved comments to land. The discipline, not the UI, is the pattern: every comment reaches a terminal state, structurally. Transplant: packet items are action items with mandatory terminal states (addressed/disputed), an unresolved-count is always visible, and "map is clean" is a definable condition — the anti-graveyard mechanism. Breaks: hard-blocking a live session on human annotation review inverts the tool (agent stalls awaiting Pi); the brief's freeze-node-on-pending-annotation should stay per-node and optional, never a global gate.

**27. Honest orphan over confident guess — Brush et al., CHI 2001 (TR-2000-95, studies read in full).** Two user studies on annotations in modified documents. Key findings: (a) orphaning was the #1 complaint — in Office 2000 Web Discussions, lost annotations were *the primary reason people stopped using the feature entirely*; (b) surprising: when the anchor text was substantially changed, participants would often have *preferred the system orphan the annotation* than reposition it plausibly-but-wrongly — satisfaction with a found annotation falls with modification count, while satisfaction with an orphan *rises* once the user sees the anchor truly vanished. Transplant: when a node is deleted or merged away, its annotations go to a visible orphan tray carrying their payload (19) — never best-guess reattached to a semantically similar node. A wrong reattachment costs more trust than an honest orphan. Breaks: an orphan tray needs its own lifecycle (dismiss/archive) and a home in the layout, or it becomes the new graveyard.

**28. Keyword/name/quote anchoring beats context — Brush et al., CHI 2001.** Second surprise: users judged re-anchoring quality by whether *key words, names, and quotations* were relocated; surrounding-context matching (the standard robust-anchoring ingredient) mattered far less than designers assumed. Transplant: when re-anchoring an annotation across node versions, match on the annotated element's distinctive terms and quotes, not on its position or neighborhood in the node body. Breaks: LLM rewrites preserve meaning while replacing exact wording — cartographer has an escape Brush's subjects didn't: schema-level stable IDs for nodes *and* intra-node claims; IDs first, keyword match as fallback only.

**29. Orphan rates justify versioned anchoring — Aturban, Nelson & Weigle (arXiv 1512.06195) + Hypothesis orphan tab.** Measured on live Hypothes.is data: ~27% of annotations already orphaned, ~61% more at risk if the page changes; only ~3.5% of orphans recoverable from web archives. Hypothesis's mitigation: a dedicated Orphans tab — failures are shown, not deleted. Transplant: the numbers are the design case for annotations referencing node@version from day one — cartographer owns its node store, so unlike Hypothesis it can *guarantee* an annotation's target version remains resolvable forever; orphaning shrinks to the delete/merge case by construction. Breaks: the guarantee holds only if node versions are genuinely immutable and never garbage-collected — a history-retention decision Phase 3 must treat as load-bearing, not housekeeping.

**30. Payload-carrying selection — Google Docs quoted comments / the W3C annotation body-target split.** The selection captures its target's content at selection time (Docs stores the quoted text; W3C annotations separate body from target with the target described by selectors + snapshot). Transplant: brief requirement made concrete — a selection snapshots node state, assumption text, linked artifacts, and provenance at select time, so the packet is self-describing even if the map moves on before submit. Breaks: snapshot and live state can diverge *during composition* — the composer should show as-selected vs current side by side the moment they differ (feeds the staleness check in 23).

**31. Re-request review closes the loop — GitHub re-request + incident.io's excluded comment layer.** GitHub: after addressing comments, the author flips the ball back with a re-review request — resolution is bilateral, not self-declared. incident.io: timeline discussion comments are structurally excluded from the postmortem export — chatter never contaminates the artifact. Transplant: when the agent acts on a packet, annotations move to "addressed-pending-confirmation" with a delta view of what changed in response; Pi confirms (or disputes) to close. And the whole annotation layer lives *outside* map content — the agent's actual fixes enter the map as normal node updates with provenance; the conversation about the map never becomes the map. Breaks: bilateral closure adds a Pi-side chore per packet — batch the confirmations (one "review the agent's response" pass per packet, not per annotation) or the closure discipline becomes closure homework.

## Need 5 — Topic boundaries and seams (incl. compaction)

**32. Frames as named containers and navigation targets — Miro frames / Obsidian Canvas groups.** A frame is a named region that owns its contents; the frame list doubles as a table of contents; presentation mode steps frame by frame. Transplant: topic nodes are containers with names, and the topic rail is the session's TOC — Mock A's "three top-level topic nodes" become three frames; a walkthrough mode steps topic by topic. Breaks: canvas frames are human-drawn; cartographer's topic boundaries are inferred and will be wrong sometimes — the design must include cheap boundary-repair gestures (merge topics, split topic, move node across seam), because a mis-drawn container poisons every view built on it.

**33. Visible seam for discontinuities — Perfetto trace-gap/clock-jump markers + CMV compaction-boundary detection.** Trace viewers render data gaps and clock discontinuities as explicit visual seams rather than smoothing over them; CMV's first trim pass exists purely to *detect compaction boundaries* in session JSONL, treating pre- and post-compaction content differently. Transplant: Mock C's compaction event renders as a horizontal seam artifact in the map — "context compacted here; nodes above anchor to pre-compaction transcript" — because provenance semantics genuinely differ across it. Breaks: a loud rupture-styled seam implies the *map* lost something — it didn't; the map is precisely what survives compaction. Style it as a provenance annotation on the spine, not a tear in the narrative.

**34. Boundary events from status transitions — MAGE subgoal boundaries + PagerDuty/incident.io status changes.** MAGE only compresses at declared subgoal completions; incident timelines auto-mark status transitions (investigating → identified → resolved) as structural events that segment the narrative. Transplant: the reducer keys topic/phase seams to detectable session events — plan-mode entry/exit, todo-list completion, explicit user pivot messages, long tool-run boundaries — and folding is only legal at seams (6). Breaks: chat pivots are softer than declared statuses; MAGE's own paper does *no* automated semantic boundary detection (boundaries are agent-declared or threshold-triggered) — so expect seam errors and design the repair gestures (32), don't assume detection accuracy.

**35. Zoom-into-subtree with breadcrumb — Workflowy zoom / org-mode narrow-to-subtree.** Clicking a bullet makes that subtree the entire page; a breadcrumb trail preserves the way back; everything else ceases to exist visually until you widen. Transplant: zooming a topic node turns a mega-session's map into a small, complete-feeling map of just that workstream — breadcrumb: session ▸ topic ▸ node. The mega-session (Mock C) becomes N small sessions on demand. Breaks: narrowing hides cross-topic edges — a plan file touched in two topics needs an "external link" indicator at the zoom boundary, or zoomed-in Pi makes decisions blind to cross-topic coupling.

**36. Block references and live embeds — Roam / Logseq.** Every block has a global ID; other locations *reference* or *live-embed* it; edits propagate to all embed sites; a backlinks panel lists every referencing context. Transplant: an artifact or decision touched by multiple topics is one node embedded in several spine positions — updated once, rendered everywhere, with backlinks answering "which topics touched this plan file?" Breaks: embeds turn the tree into a DAG, and a DAG has no single top-to-bottom reading — cartographer needs a primary-home rule (node lives in one topic, others get reference stubs) to keep the spine linear.

## Need 6 — Density and overview navigation for long sessions

**37. Minimap with brush — Perfetto/Jaeger overview strip, VS Code minimap.** A compressed strip of the entire artifact shows density/structure at a glance; a draggable brush selects the visible window; you always know where you are and how much exists. Transplant: a session minimap rail — spine extent with topic-color bands and markers for decisions, seams, and open annotations; brush to jump; in live view the strip grows at one end. Breaks: trace minimaps encode wall-clock time; cartographer's must encode *spine order* (reading position), not the clock — a time-axis minimap quietly re-imposes the log.

**38. Degree-of-interest rendering — Furnas fisheye (1986), Card & Nation DOI trees, SpaceTree.** Node detail is a function of distance from the focus: near = full, far = stub; the layout auto-compresses uninteresting regions so arbitrarily large trees fit a screen. Transplant: at rest, the map renders full detail around current activity (live) or last focus (post-hoc) and auto-stubs distant *completed* topics — mega-sessions stay one screen tall without manual folding. Breaks: DOI reflow fights spatial memory — nodes move and resize as focus shifts; without animated transitions and stable landmarks (pinned topics, the minimap), Pi loses his sense of place, which for a visual person kills the tool.

**39. Pinning + auto-expiring seen-state — Perfetto pinned tracks, GitHub file "Viewed" checkbox.** Pin rows of interest to a persistent header; separately, per-viewer "viewed" marks collapse content and *automatically clear when the content changes*. Transplant: Pi pins open questions and key decisions to a persistent rail regardless of scroll; per-node seen-marks collapse reviewed regions and self-clear on version bump — combined with (11), "what needs my eyes" becomes computable. Breaks: none structural; requires per-viewer state stored outside the map artifact so the map itself stays viewer-independent.

**40. Width-as-weight — flame graphs.** Visual size is proportional to cost, so the expensive path is visible pre-attentively, without reading a single label. Transplant: encode per-node effort (turns/tokens consumed) visually, so a glance shows where the session's effort went — the false-alarm detour renders thin, the real debugging arc thick. Breaks: in conversation, effort ≠ importance — a one-line decision can outweigh fifty turns of yak-shaving; structural size-by-effort would misrank the map's content, so effort likely belongs in a subtle channel (badge, gutter bar), not in layout-driving node size.

**41. Semantic zoom — Prezi, modern canvas tools, map applications.** Zoom changes *representation*, not just scale: far out shows only region labels, mid shows titles, close shows full content — each zoom level is a designed view. Transplant: one continuous gesture spans day-view → topic labels → node faces → quote-level detail; density management without any folding UI at all. Breaks: each zoom level is its own layout to design and keep coherent (multiplies Phase 1–2 design cost), and continuous zoom breaks the "prints clean top-to-bottom" document property — plausible for a canvas-paradigm direction, wrong for a document-paradigm one.

**42. Entity-view separation — Heptabase cards / whiteboards.** A note is one identity; whiteboards are views that *display* it; the same card appears on many boards, edited anywhere, updated everywhere — views are cheap, data is singular. Transplant: nodes are identities; the spine, topic zoom, and any future day/multi-session views are projections over the same node store — later views (fenced to post-v0) become rendering work, not data-model work. Breaks: nothing at schema level; the danger is treating it as license to build multi-view UI in v0 — the brief's Wave fence says exactly not to.

**43. Chapter markers on a scrub bar — YouTube chapters / podcast chapters.** Long linear media gets named waypoints rendered as segment boundaries directly on the scrub bar; hovering previews, clicking jumps; the bar itself communicates macro-structure. Transplant: topic boundaries and gates render as labeled ticks on the minimap/scroll rail — the rail alone tells the session's story ("3 topics, one seam, 2 open questions") before any node is read. Breaks: chapters assume a single linear axis — fine for the spine, but ticks must survive folding/zooming (positions are logical, not pixel-fixed).

## Anti-patterns observed — things adjacent tools do that cartographer must NOT copy

- **Hidden-state divergence as normal operation (Jupyter).** The notebook's displayed document routinely lies about runtime state (out-of-order execution, deleted cells with live variables), and the ecosystem shrugs. Cartographer's equivalent — map content silently diverging from transcript truth — is a trust-extinction event; the validate pass (14) exists because this failure mode is documented, not hypothetical.
- **Outdating on any touch (GitHub file-level comments — discussion #86527).** Marking annotations outdated on *every* push, including changes irrelevant to the annotation, trains users that "outdated" is noise. Outdate on semantic change to the annotated element only; a cosmetic reducer rephrase must not bump the version annotations anchor to.
- **Silent orphaning / orphans dumped at the document bottom (early web annotation systems, per Brush et al.).** Lost annotations were the #1 reason users abandoned Office 2000 Web Discussions. Orphaning without a visible state, payload, and lifecycle is feature abandonment on a timer.
- **Confident wrong reattachment (naive robust-anchoring).** Brush's users preferred honest orphans to plausible-but-wrong repositioning. Never reattach an annotation to a "similar" node by guess.
- **Porting all history forward (Gerrit 3.5 ported comments, per user complaints).** Auto-surfacing every unresolved item from every past version onto the current view silts the interface. Port unresolved only, and let resolution — not survival — be the default lifecycle exit.
- **Global graph view as the primary surface (Roam/Obsidian graph mode).** The all-nodes force-directed graph demos beautifully and answers no narrative question. The spine is the product; graph-like views are, at most, a lens.
- **Expert-density walls (Perfetto/Jaeger at rest).** Trace viewers assume a trained operator and open on maximal density. Cartographer's at-rest render must be readable by tired-end-of-day Pi with zero interaction; density is opt-in (3, 38).
- **Chronology as the organizing axis (every log-faithful session tool).** The entire positioning: time-ordered rendering is the thing being replaced. Chronology is a lens (20), a provenance fact (17), and a minimap annotation — never the spine's sort order.
- **Version-noise notification fatigue (Figma/Docs minor-version churn).** Surfacing every micro-revision as a visible event teaches users to ignore all change signals. Auto-collapse mechanical versions (10, 12) so the glint (11) stays meaningful.
- **Hybrid-everything surface (Google Wave — already fenced in the brief).** The research here (42) makes multi-view *cheap later*; that is an argument for schema discipline now, not for building day-views in v0.
