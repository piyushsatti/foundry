# Canvas Framework Research — Foundation for Cartographer (and future canvas tools)

Date: 2026-07-11. Evaluation lens (per Pi's refinement): the winner must be **dual-nature** — (A) powerful for a human on an infinite canvas (drag, link, arrange, Heptabase feel) AND (B) a well-structured data layer with stable IDs, a queryable/mutable store, and change subscriptions so Claude can operate on it via MCP or a thin service. Data-model quality and bidirectional sync are weighted above raw canvas features.

## Executive recommendation

**Top pick: React Flow (`@xyflow/react`)** — MIT-licensed, 36.4k+ stars, actively maintained, and the only candidate where the entire document is *by construction* a plain-JSON graph you own: `nodes[]` and `edges[]` with stable string IDs, controlled React state, granular change events (`onNodesChange`, `onConnect`), and edges that auto-attach to node handles and re-route on every drag. Claude writes JSON, the human drags, and the exact same arrays flow back out — there is no impedance mismatch because there is no hidden internal format, which makes it the strongest reusable foundation for multiple agent-facing tools. **Runner-up: Excalidraw** (MIT, 127k stars) — the best human-feel canvas that is genuinely open source, with a documented JSON scene format, element-level arrow bindings, an imperative `updateScene`/`onChange` API, and an existing MCP ecosystem (official `excalidraw/excalidraw-mcp` plus a 26-tool community server with live WebSocket sync); its weakness is that the scene is a freeform drawing, not a typed graph, so semantic structure must be layered on via `customData`. tldraw would rank first on pure dual-nature merit (best-in-class reactive record store) but fails the hard open-source requirement — it is source-available under the proprietary "tldraw license" with a watermark/license-key regime.

## Per-candidate notes

### React Flow (`@xyflow/react`)
The de-facto standard React library for node-and-edge UIs (n8n, Stripe docs, Typeform, Langflow-class tools are built on it). Building on it means: you define custom node components (any React — cards, markdown, images), keep `nodes`/`edges` in your own store (Zustand is the maintainers' recommended pattern), and get pan/zoom, grid background, snap-to-grid, drag, multi-select, minimap, and live edge re-routing out of the box. Killer strength for us: **the document IS the API** — stable-ID JSON arrays, granular change events, no import/export ceremony, so an MCP layer is an afternoon of work, and the same skills transfer to every future tool. Watch-outs, not dealbreakers: default edge routing is simple bezier/smoothstep (obstacle-avoiding routing needs a lib or Pro example), and "whiteboard feel" (freehand, ink) isn't its idiom — it renders crisp cards, which matches cartographer's session-map use case anyway. MIT with an optional paid Pro subscription (examples + support only; the library is complete without it).

### Excalidraw
A virtual hand-drawn-style whiteboard shipped as an embeddable MIT React component with a documented plaintext JSON scene format (`.excalidraw`). Building on it means embedding `<Excalidraw excalidrawAPI={...} onChange={...}>` and programmatically constructing elements via `convertToExcalidrawElements` skeletons; arrows carry `startBinding`/`endBinding` element references so they stay attached as the human drags cards. Killer strengths: the largest community of any candidate (127k stars), genuinely delightful human UX, and the **most mature MCP story** — an official `excalidraw/excalidraw-mcp` plus `yctimlin/mcp_excalidraw` (26 tools, element-level CRUD, agent-readable screenshots, WebSocket sync to a live browser canvas). Dealbreaker-adjacent: the model is a *drawing*, not a *graph* — no typed node schema, semantic meaning must be smuggled through `customData`, and heavy programmatic mutation fights the internal versioning/fractional-index machinery more than React Flow's plain arrays do. Bundle is large and asset-laden, so strict-CSP single-file HTML is possible but painful.

### tldraw
The best-engineered canvas SDK in the ecosystem: reactive typed record store, `store.listen` for granular change streams, `getSnapshot`/`loadSnapshot`, and tldraw sync for multiplayer — exactly the dual-nature architecture we want. Disqualified by license: since v2/v3 it is **not open source** — the "tldraw license" permits free use in development only; production requires a license key (free hobby tier with a mandatory, non-removable "made with tldraw" watermark; paid Business license to remove it), and tampering with key validation or the watermark is prohibited. That's a rent-seeking dependency at the foundation of a multi-tool platform, and the phone-home key validation sits badly with self-contained offline HTML. Treat it as design inspiration, not a foundation.

### AFFiNE / BlockSuite
AFFiNE is the open-source Notion+Miro hybrid (70.3k stars) — the closest full-app analog to Heptabase — with docs and an "edgeless" canvas over one Yjs CRDT document model; editor and most code are MIT, the server is under an Enterprise Edition license. BlockSuite, its editor framework, is nominally reusable ("other projects should equally reuse and extend BlockSuite"), and its data layer is the strongest on paper: typed blocks, stable IDs, CRDT sync, time-travel. In practice BlockSuite now lives as a 60+-package sub-monorepo inside AFFiNE, standalone packaging/docs have decayed, and betting cartographer on extracting it is a research project, not a foundation. Verdict: integrate *with* AFFiNE later if Pi wants a knowledge-base home for maps; don't build *on* BlockSuite today. No established AFFiNE MCP server surfaced in research.

### JointJS
The oldest serious diagramming library (since 2010), vanilla JS, with a real graph model (`graph.toJSON()`, cell IDs, events) and proper link routing including manhattan/orthogonal — arguably the best *connector* engine among the open options. Gotcha: open-core — the MPL 2.0 core is capable, but the polished widgets, advanced routing behaviors, and UI chrome live in commercial JointJS+, and the vendor's docs steer you there constantly. MPL 2.0 is file-level copyleft (fine for our use, but not MIT-clean). Community is far smaller than React Flow's. A reasonable fallback if we ever outgrow React Flow's routing and want framework-agnostic vanilla JS.

### Rete.js
MIT, 12,089 stars, active. It's a visual *programming* framework — nodes with typed input/output sockets and a dataflow/control-flow engine — aimed at "boxes that compute", not "cards you think with". The plugin architecture (area, connection, render plugins per framework) gives flexibility but means assembling many pieces to reach a baseline React Flow ships by default. Wrong center of gravity for a Heptabase-like map; right tool if cartographer ever needs executable pipelines.

### Konva / react-konva
MIT, active, excellent at what it is: a retained-mode 2D canvas scene graph (shapes, drag, events, transforms) with React bindings. It has an `Arrow` shape but **no document model, no connector semantics, no pan/zoom/grid framework** — every hard requirement here is a DIY project. Choose it only if we wanted total rendering control and were prepared to write our own mini React Flow. For cartographer that's negative leverage.

### Logseq / Drawflow / litegraph.js (short)
**Logseq**: AGPL-3.0 full app; its datascript-queryable DB and open markdown format are genuinely agent-friendly, and Whiteboards exist (built on a tldraw fork), but it's an app to integrate with and AGPL blocks reusing its code in our tools. **Drawflow** (MIT, 6,047 stars) and **litegraph.js** (MIT, 8,059 stars) are small vanilla-JS node editors with JSON export — both effectively stale upstream (litegraph survives mainly via ComfyUI's fork). Fine for one-offs; wrong for a platform.

### JSON Canvas (format, not engine)
Obsidian's open, MIT-licensed infinite-canvas file format (v1.0, March 2024): `nodes` (text/file/link/group) and `edges` with stable IDs, sides, and labels. Deliberately conservative, designed for interoperability. Recommendation independent of engine choice: **make `.canvas` (JSON Canvas) cartographer's on-disk interchange format**, with a lossless superset for React-Flow-specific fields — maps then open in Obsidian for free, and any future tool in the family shares the format.

## Bidirectional sync sketch (the heart of it)

### React Flow — Claude writes / human drags / state flows back

The store is ours, so sync is a data-plumbing problem, not a framework fight.

- **Single source of truth**: a Zustand (or plain reducer) store holding `nodes: Node[]` and `edges: Edge[]` — exactly the JSON Claude reads and writes. Persist as JSON Canvas on disk (`map.canvas`), with React-Flow extras (handles, node type, dimensions) under a namespaced key.
- **Claude → canvas**: an MCP server (or the cartographer plugin's local service) exposes `add_node`, `update_node`, `link`, `set_layout`, `read_map`. Each mutation edits the JSON file / store; the running web app picks it up via a WebSocket push or `fs.watch` → `setNodes`/`setEdges`. React Flow re-renders; edges referencing the node IDs re-route automatically. For a no-server variant, the self-contained HTML can poll/import a JSON blob pasted or loaded from disk.
- **Human → state**: React Flow emits granular intents — `onNodesChange` (drag/position/select/remove), `onEdgesChange`, `onConnect` (human draws a new arrow between handles). Apply with `applyNodeChanges`/`applyEdgeChanges`, debounce, then `rfInstance.toObject()` → `{nodes, edges, viewport}` → serialize back to `map.canvas`. Positions the human chose are just `node.position` fields in the same JSON Claude reads next turn.
- **Conflict policy**: last-writer-wins per node ID is fine for a single human + single agent; if it ever needs real concurrency, wrap the store in Yjs (`y-websocket`) — community bindings for React Flow + Yjs are established.
- **Why this wins**: there is no translation layer. The thing Claude mutates, the thing the human drags, and the thing on disk are the same three arrays.

### Excalidraw — Claude writes / human drags / state flows back

- **Claude → canvas**: two ready paths. (a) File-centric: Claude writes/edits a `.excalidraw` JSON document (documented schema; elements have stable `id`s, arrows carry `startBinding`/`endBinding` to element IDs); the app loads it via `excalidrawAPI.updateScene({ elements })`, building elements with `convertToExcalidrawElements` so bindings/versioning are valid. (b) Live: run `yctimlin/mcp_excalidraw` — MCP tools do element-level CRUD against a canvas server, and a WebSocket pushes every change into the open browser canvas in real time; it also gives Claude `describe`/screenshot tools to *see* the board.
- **Human → state**: `<Excalidraw onChange={(elements, appState) => ...}>` fires on every user edit — drag, resize, new arrow, delete. Debounce, run `serializeAsJSON(elements, appState)`, write back to the `.excalidraw` file (or POST to the canvas server). Because arrows are *bound* to elements, human rearrangement preserves the connection topology Claude created, and the bindings in the JSON tell Claude what connects to what.
- **Semantics layer**: Excalidraw elements have no "node type", so cartographer's meaning (session, event, artifact...) goes in each element's `customData` — Claude filters/queries on it; the human never sees it.
- **Trade-off vs React Flow**: you get the superior hand-drawn human feel and a pre-built MCP ecosystem, but Claude must construct valid Excalidraw element records (version, seed, fractional index, binding objects) rather than three-field JSON nodes, and queries operate on a flat drawing rather than a typed graph. More power on day one (MCP exists), more friction on day one hundred (the model isn't yours).
