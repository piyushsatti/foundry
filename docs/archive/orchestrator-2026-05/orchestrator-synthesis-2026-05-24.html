§1 Your vision (verbatim)

Orchestrator is like an advanced version of Airflow or like an advanced version of a React loop. It's basically a sort of engine that is also state machine but also tracks what's happening at different levels. It has a DB. It has MCP tools. It can query, like, what's at what state. It can present to me, like, what's running, how much of it's done, what's next, what's the quality of the different code or different PRs that we have written, what's the iteration level.

I should feel like a CTO that's sitting in this, like, massive office having tabs on all the different projects. As if I'm sitting on this, like, cockpit, and I'm having a visual of everything that's going on. We can slow certain things down. I can fast forward other things.

Failure is not seen as something that's to be avoided. Unilateral decisions should be seen as something to be avoided. The subagents [should] know that it's okay and feasible and wanted to surface issues that they are facing. — you, 2026-05-24

This vision has four distinct loads in it: autonomy (orchestrator does work without you), visibility (you see everything that happens), control (you can speed up / slow down / intervene), and honest failure (subagents surface issues rather than guess). Each subagent below stress-tested a different leg of that.

§2 The four perspectives

1. The systems engineer substrate · correctness · recovery

"You will regret this within a week of running it."

Came in tearing apart the state model. Built workflow engines before; doesn't believe in vibes-as-design. Pushed hardest on:

  • The 5-state model conflates where the job is in its lifecycle with what its outcome was. Crash recovery needs the first; reporting needs the second. They're orthogonal.
  • Missing states: timeout (single most common AI failure mode), crashed (≠ failed), blocked (parent drift detected pre-run), needs_review (the explicit "ask-early" success state).
  • Fencing tokens on every writeback from day one (Kleppmann pattern). Without it, retried writebacks corrupt the manifold spec.
  • "Fail loud, ask early" needs technical teeth — mandatory output schema with assumptions_made[]; auto-downgrade achieved → needs_review if non-empty. Make lying expensive, asking cheap.
  • Phase rows (one per phase entry) are the recovery unit, not job rows. Phase 2 hooks just become more phase rows. Phase 3 parallelism is N workers writing phase rows. The substrate doesn't change.

One direct disagreement with your framing: "failure not to be avoided" is correct as philosophy but dangerous as implementation default. Distinguish failure-as-signal (subagent said failed with evidence — keep) from failure-as-noise (timeout, crash, schema violation — retry or escalate, don't pollute the dataset).

2. The cockpit UX designer surfaces · hierarchy · honest signals

"The 'CTO in a massive office' framing is wrong, and you should drop it."

Designed dashboards for trading desks, SRE consoles, render farms. Came in correcting the metaphor before answering anything:

  • Drop "CTO cockpit." A CTO watches people; you're watching jobs. The right reference is an SRE on-call dashboard or a render-farm queue console. Same calm-comprehensive-in-control feeling; cleaner design constraints.
  • The dashboard must answer three questions in 2 seconds of a glance: Is it still running? Is anything stuck? What got done since I last looked? Everything else is drill-down.
  • "Speed up / slow down" is cinematic but vague — the real primitives are pause job, cancel job, bump model tier (haiku→sonnet→opus), demote model tier, re-queue with note, drag-to-reorder, concurrency slider, daily budget cap.
  • Quality signals split sharply into real (tests, lint, files, tokens, manifold validation, tool-call rounds), partial (LLM judge scores — show next to tests, never alone), and vapor (aggregate "code health," velocity graphs, ETAs — do not build).
  • "Iteration level" you mentioned: needs to be precisely defined or dropped. Three meanings overlap and none is honest as a single number.
  • The daemon heartbeat is sacred. Top-left of every page, always visible, red and loud if >60s stale. Without it, every other signal on the page is suspect.

3. The SRE / observability engineer logs · diagnostics · silent failures

"It's 9am. Operator opens cockpit. Sees one red row. Clicks it."

Has been paged at 2am too many times. Believes observability is a property of the system, not an add-on. Pushed three specific designs:

  • Three log streams, not one. Event log (structured, in DB, every phase transition). Session transcripts (JSONL per job, in files). Daemon log (structured JSON, non-job-tied events). Never grep a 12GB file at 2am.
  • Verdicts must return structured failure data, not booleans. "Verdict failed" tells you nothing. "Verdict failed: expected=200, actual=500, diff=…" tells you everything. This is a manifold contract change.
  • Silent failure detection. Three guards: (a) phase-completeness invariant — every "success" must have all phase rows or it's a silent failure; (b) writeback verification — read back after write, assert manifold changed as expected; (c) staleness check on verdict — record manifold revision at claim AND at verdict, alarm if they don't match.
  • Subagent "guessing" is partially observable. Free signal: count clarifying questions, mutate-vs-read tool-call ratio. Built signal: typed issue_raised events (ambiguity / missing_context / conflicting_constraints) as first-class tool calls. Proxy: post-hoc verdict failure rate per subagent.
  • Drift detection should be both subagent-reported and orchestrator-caught — tagged differently (drift_reported_by = subagent vs orchestrator) because they mean different things diagnostically.

4. The red-team thinker first principles · assumptions · "dumbest version that works"

"Build the bash script first. The 'advanced Airflow' framing is aspirational cosplay."

Doesn't soften. Made the sharpest call of the four:

  • "You don't have an Airflow problem. You have a 'I want to walk away from my computer for 2 hours' problem." Airflow exists because hundreds of analysts share one scheduler. You are one person on one laptop running maybe 5–50 leaves a day.
  • The dumbest version that could work: a bash while read loop calling Claude + a verifier, logging to a file. 80% of value, zero infrastructure. "Phases 2 and 3 of v0.1 are not v0.1. They are v0.3 and v0.5 disguised as 'phases.'"
  • Five killer assumptions to challenge:
    • "Manifold is single source of truth" — no, it's a recorded truth. Drifts the moment a human edits a file outside the loop.
    • "Verdict_check accurately measures done" — false passes are the dominant LLM-verification failure (Claude will write a stub test that passes; overfits to test cases).
    • "Subagents surface issues honestly" — LLMs are sycophantic; they surface what THEY find interesting, weighted toward the obvious, rephrased in agreeable language.
    • "Claude is the right router for skill selection" — no feedback loop; silent miscategorization is the default.
    • "Daemon model fits this workload" — daemons are for continuous load; yours is bursty and solo-triggered. orchestrator run invoked when you walk away is simpler.
  • Failure modes nobody planned for: SQLite database-is-locked during writeback; user edits leaves while daemon is processing them (no etag at claim); non-idempotent verdict_checks fire twice on crash (verifier that opens a GitHub PR creates duplicates); runaway cost burning $50/hr while you sleep; "step-away gap" where leaf 1 destructive migration + leaf 2 depends-on-old-schema = broken repo with green dashboard.
  • The cockpit will lie: green when integrated system is broken (per-leaf verdicts don't check the joint); progress when looping (retry-to-success looks like first-try); productivity when wasteful (47 trivial doc edits looks like work); calm when on fire (cached dashboard during outage).
  • Architectural debt risk: calling it a "daemon" commits you to uptime / supervision / log rotation / signal handling forever. "Call it a runner invoked on demand and you keep the option to upgrade to daemon later. The reverse path is much harder."

§4 Where they fight — and my adjudication

Disagreement 1 — "CTO cockpit" framing

UX + Red Team converge: wrong metaphor. UX explicitly: "A CTO watches people; you're watching jobs. The right reference is an SRE on-call dashboard or a render farm." Red Team agrees: "Solo devs want to NOT LOOK AT THINGS."

Adjudication: drop the "CTO cockpit" framing. Keep the user-facing intent (calm, comprehensive, in-command). Adopt "SRE on-call console" or "render-farm queue" as the design metaphor. Both surface only what needs attention; both have heartbeats as load-bearing signals; both treat silence as a property to design around, not a failure to fix.

Disagreement 2 — Separate DB or shared with manifold?

Systems Engineer says SEPARATE (Airflow pattern; different concerns, different schema velocities; cross-DB integrity via messages, never joins). Red Team says SHARED (two-database consistency is forever).

Adjudication: separate DB. Red Team's pragmatic concern is real but the answer is the Systems Engineer's contract — fencing tokens + idempotency keys on every cross-DB write. We need that contract for crash recovery anyway. Coupling the schemas would mean every orchestrator migration risks the spec store, which is strictly worse.

Disagreement 3 — Daemon or runner?

Systems Engineer, UX, SRE all assumed daemon. Red Team pushed back hard: "Daemons are for continuous load. Your load is bursty and solo-triggered. make run-leaves invoked when you walk away is simpler and has zero uptime semantics to debug."

Adjudication: Red Team wins. The "step away and trust" affordance you stated does NOT require a daemon — it requires a long-running invocation that exits when done. The daemon model adds real ops cost (systemd / launchd, signal handling, log rotation, supervisor) for no marginal benefit at solo workload. Build it as orchestrator run; promote to daemon only if Phase 3+ ever needs continuous polling. Calling it "runner" preserves the upgrade path.

Disagreement 4 — Bash script vs. structured Phase 1

Systems Engineer says 6-table substrate from day one. Red Team says bash script first; let real pain dictate Phase 1.

Adjudication: add a Stage 0 before Phase 1. Two-day bash spike against chronicler. Document what hurts. THEN commit to Phase 1 design. The substrate we'll likely build IS the Systems Engineer's, but the spike tells us if any of it is theatrical. Cheap insurance against overdesign.

Disagreement 5 — Hooks contract in Phase 1?

Systems Engineer: scaffold the table (empty, ready for Phase 2). Red Team: plugin APIs are forever — don't ship them until you know what hooks you need.

Adjudication: Red Team. Write lifecycle code with explicit named phases. If Phase 2 needs declarable hooks, extract from the existing code at that point. Cheaper to extract than to design-without-evidence.

§6 Concrete good-vs-bad examples

These are the load-bearing concepts where the perspectives sharply agreed. Each pair shows the wrong way (which is also the seductive way) and the right way side-by-side.

State machine

⚠ Bad — collapsed state

Single axis, conflates lifecycle and outcome

queued → claimed → running →
  {achieved, failed, abandoned, drift}

When a job goes from running to failed: was it a subagent that found a real issue and reported failure (signal — keep, learn from), or was it a daemon crash mid-session (noise — retry)? The state doesn't distinguish. Your retry logic conflates them; your metrics lie.

✓ Good — two orthogonal axes

Lifecycle (where) × Outcome (what)

Lifecycle: queued → claimed → pre_hooks
  → running → verdict → post_hooks
  → writeback → released → terminal

Outcome (at terminal):
  achieved | failed | abandoned | drift |
  crashed | timeout | cancelled | needs_review

Recovery scans for stale heartbeats per lifecycle state (idempotent retry vs needs compensation). Reporting groups by outcome. Different concerns, different axes. Adding timeout and crashed as separate outcomes from failed stops poisoning your dataset.

Verdict return shape

⚠ Bad — boolean

You see this in the cockpit and have to grep

{ "status": "violated" }

9am. You open the cockpit. You see one red leaf. You click it. You see: "violated". Now you SSH to find the log file and grep for what actually failed. The cockpit failed to help you.

✓ Good — structured failure data

You see this and you immediately know

{
  "status": "violated",
  "expected": "POST /checkout returns 200",
  "actual": "POST /checkout returns 500",
  "diff": "stack trace: NullPointerException at...",
  "evidence": [
    "test_logs/checkout.log:42",
    "fixtures/valid_order.json"
  ]
}

Operator opens the leaf, sees expected vs actual at a glance, has a path to debug. Doesn't have to leave the cockpit.

The cockpit lying about state

⚠ Bad — false confidence

Looks great, daemon died 20 min ago

┌──────────────────────────────┐
│ ORCHESTRATOR  ✓ HEALTHY     │
│ Project: 47/89 leaves done  │
│ All systems nominal         │
└──────────────────────────────┘

The dashboard is cached. The runner crashed 20 minutes ago. Nothing has updated. The green badge is a lie. You walk away confident.

✓ Good — heartbeat as load-bearing signal

If the heartbeat is stale, every other signal is suspect

┌──────────────────────────────┐
│ ORCHESTRATOR [RUNNING 4h12m] │
│ last heartbeat: 14s ago      │
│ ─────────────────────────────│
│ ATTENTION (0)  All clear     │
│ NOW RUNNING (2) ...          │
└──────────────────────────────┘
heartbeat ages: green < 30s,
                amber 30-60s, RED > 60s

Heartbeat top-left, always visible. Red and loud if >60s. Without it, every other signal might be from before the system died.

Daemon vs runner

⚠ Bad — daemon (premature)

Continuous uptime semantics, solo workload

# systemd unit, pidfile, log rotation,
# signal handlers, restart policy,
# port conflict handling, "why is the
# daemon eating CPU at 3am" debugging

$ orchestrator serve --pidfile X --log Y
$ systemctl --user enable orchestrator

You're buying uptime infrastructure for bursty solo work. Every crash is a 3am page even though nobody is going to be paged.

✓ Good — runner (right-sized)

Invoke when you walk away. Exit when done.

# Just a long-running invocation.
# Exits when queue empty, budget hit,
# or time up. No supervision.

$ orchestrator run \
    --project chronicler \
    --max-budget $10 \
    --max-time 4h
[runner exits when one of:
 - queue empty
 - $10 spent
 - 4 hours elapsed]

If you decide Phase 3 needs continuous polling, promote to daemon then. Runner → daemon is easy. Daemon → runner is a rewrite.

"Fail loud, ask early" — wire-format teeth

⚠ Bad — stated as principle in prose

skill.md says "please surface issues"

# In orchestrator's skill.md
"If you have any uncertainty
about what to build, please
surface it rather than guess."

# In the DB, 3 months later:
SELECT change_reason, count(*)
FROM revisions
GROUP BY change_reason;

evolution   847
drift         0   ← agents learned to lie

Agents read the principle, ignore it, write achieved and move on. The principle has no teeth.

✓ Good — schema-enforced auto-downgrade

Lying becomes expensive at the wire level

# Required output schema for any subagent:
{
  "verdict": "achieved" | "failed"
           | "blocked"  | "needs_human",
  "evidence": [string],
  "assumptions_made": [string],
  "unresolved_questions": [string],
  "change_reason": string
}

# Orchestrator's auto-downgrade rule:
if verdict == "achieved"
   and assumptions_made.length > 0:
   verdict = "needs_human"
   # Surfaces in cockpit ATTENTION panel
   # Costs the agent its "done" claim

Silent assumptions become a protocol violation the engine catches. "The orchestrator's job is to make lying expensive and asking cheap."

Architecture — manifold ↔ orchestrator

The two skills don't talk via DB joins. They talk via the CLI/MCP contract. Each owns its schema; cross-system writes carry idempotency keys.

§7 Proposed v0.1 contract

If you accept the reframes in §5, the v0.1 substrate locks at this. It's smaller than the original proposal because the runner reframe (vs daemon) eliminates substantial complexity, and because hooks/web-UI move to later phases informed by real usage.

Stage 0 — bash spike (2 days)

Literal bash script orchestrator running against chronicler. while read leaf; do claude -p ...; verifier; manifold update-node; done. Document what hurts. Phase 1 design informed by spike findings.

Phase 1 — informed minimum (weeks)

Substrate

  • orchestrator is a CLI runner, not a daemon (orchestrator run --project X --max-budget $N --max-time Yh)
  • Separate SQLite DB at ~/.claude/orchestrator.db, WAL mode, busy_timeout 5000
  • Six tables: jobs, job_phases, leaf_leases, writeback_log (idempotency_key PK), events, schema_migrations
  • Two orthogonal state axes (lifecycle × outcome) — see §6 diagram
  • Phase rows append-only, one row per phase entry, with heartbeat — recovery unit

Fail-loud enforcement

  • Mandatory structured output schema for every subagent (verdict, evidence[], assumptions_made[], unresolved_questions[], change_reason)
  • Auto-downgrade rule: achieved + assumptions_made[] non-empty → needs_review
  • Typed issue_raised events (ambiguity / missing_context / conflicting_constraints / out_of_scope) — counted per-subagent
  • Wall-clock timeouts per phase (20m running, 2m others) — daemon-enforced, not agent-enforced

Observability

  • Three log streams: event log in DB · session transcripts per-job in JSONL files · runner log file
  • Writeback verification: read-back after write, assert manifold changed as expected
  • Phase-completeness invariant checker (runs periodically; alarms on success-without-all-phase-rows)
  • Heartbeat freshness as load-bearing signal in CLI status / future cockpit

Surface

  • CLI only: orchestrator run · orchestrator status · orchestrator logs <job_id> · orchestrator retry <job_id> · orchestrator cancel · orchestrator version
  • No web UI in Phase 1. No MCP server in Phase 1. These are Phase 2/3.

Manifold contract changes required (flag for v0.4.1)

  • Verdict mechanism returns structured failure data, not boolean
  • update-node accepts fencing tokens, rejects stale ones (Kleppmann pattern)
  • update-node accepts idempotency keys, dedupes

These are additive changes (don't break existing data). ~1-2 days estimated. Must ship before orchestrator Phase 1 work or orchestrator's correctness guarantees can't be met.

Quality signals — ship REAL only

  • Ship: tests pass/fail counts, lint, files touched, tokens, wall-clock time, manifold validation, tool-call rounds, retries-per-leaf
  • Do not ship: aggregate "code health" gauge, velocity graphs, predicted ETAs, single-number "project quality" scores

§8 Three decisions only you can make

Q1 — Do you accept the runner-not-daemon reframe?

Big change from your stated vision ("daemon", "cockpit", "advanced Airflow"). The reframe keeps the intent (autonomous work, step-away affordance) but discards the infrastructure assumption. If you want a true daemon for non-trivial reasons (e.g., "I want this running 24/7 polling for new leaves even when I'm not at my computer"), say so and we keep daemon — but acknowledge the ops surface that comes with it.

Q2 — Do you accept Stage 0 (the bash spike)?

Two days of running a literal bash script before committing Phase 1 code. Cheap insurance against overdesign. Or: skip the spike, trust the synthesis, commit to Phase 1 design now.

Q3 — Do you authorize the manifold v0.4.1 contract changes?

Structured verdict failure data + fencing tokens + idempotency keys are needed for orchestrator to work safely. They're additive (don't break existing data) but they're real manifold work. If yes, we plan it as v0.4.1 patch before orchestrator Phase 1; if no, orchestrator builds against current manifold and accepts substantial correctness risk.
