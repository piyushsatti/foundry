# The Sensible Way to Run Claude Code on a Personal Laptop and Drive It From Your Phone

## TL;DR
- **The converged "sensible" stack is SSH + tmux + Tailscale, accessed from a mobile SSH client (Termius or Blink Shell), with mosh added for connection resilience and ntfy push notifications so you know when the agent needs you.** This is what virtually every 2025–2026 developer write-up lands on, and it works on any Claude Code plan for $0 beyond your existing subscription.
- **For your specific complaint — buggy voice in the Claude Code mobile app — the preferred middle-grounds are Happy Coder (free, open-source; its voice is a "cleanup agent" that turns rambling into clean prompts) or, if you stay in a terminal, Secure ShellFish's "Type Sentence"/on-device dictation mode, which is explicitly designed for "writing instructions for Claude Code."** Anthropic's own native CLI voice does not work over SSH at all (needs a local mic), which is why phone users must reach for one of these.
- **For browsing/viewing files (especially markdown plan files) from your phone, the cleanest option is Secure ShellFish mounting your laptop into the iOS Files app over SFTP; for richer review, run code-server or `opencode`/VibeTunnel and expose it on your tailnet with `tailscale serve`.** Use Tailscale's own features (Tailscale SSH, Serve, Taildrop, exit nodes) selectively — most are relevant to you, Funnel mostly is not.

## Key Findings

### 1. There is a real, documented consensus on "the sensible setup"
This is not a fringe workflow. There is a remarkably consistent body of 2025–2026 write-ups — Pete Sena (Medium), skeptrune.com, Elliot Bonneville, esc.sh, Quan Nguyen, Sameer Halai, Kareem F., and others — all independently converging on the **same five primitives**: an always-on host machine, Tailscale for networking, SSH for the connection, tmux for session persistence, and a mobile SSH client. As skeptrune.com puts it bluntly: *"Every startup trying to solve 'Claude Code on mobile' is building abstractions on top of these primitives. They're not giving you anything you can't already do with SSH and Termux."*

The killer property everyone emphasizes is that **tmux lets you attach to the *same running Claude Code process*** from phone and laptop — no `/resume`, no context reload. Elliot Bonneville: *"with tmux, you're attaching to the same running process — Claude Code is still right there, mid-thought, no resume needed."*

### 2. Anthropic now ships an official option — but it has real limits
On February 25, 2026, Anthropic launched **Remote Control** (`claude remote-control` or `/rc`). Your local session keeps running on your machine; the phone/web is "just a window." It makes only outbound HTTPS calls (no inbound ports) and uses multiple short-lived, independently-expiring credentials over TLS. Per Anthropic's Claude Code docs, Remote Control "is in research preview and available on all plans. On Team and Enterprise, it is off by default until an admin enables the Remote Control toggle"; the @claudeai launch post (Feb 24, 2026) said it was live "in Research Preview for Max users, and coming soon to Pro users." (Note: the docs now list all plans while early coverage emphasized Max/Pro — a discrepancy worth being aware of as the preview evolves.) API keys are not supported.

**The catches that matter for you:** the terminal must stay open and the machine awake; per the Remote Control research-preview notes (claudefa.st, citing Anthropic docs), "One session at a time: Each Claude Code instance supports one remote connection"; and per Anthropic's docs and DevOps.com (Feb 2026), "If the network stays down for more than roughly 10 minutes, the session times out... Run claude remote-control again to start fresh." Simon Willison also noted in initial testing that session crashes can surface as "mysterious API errors" rather than clear termination messages. The DIY tmux stack has no such timeout — tmux sessions persist indefinitely. So Remote Control is the zero-setup quick-check option; the SSH+tmux stack is the resilient power-user option. Many people run both.

### 3. Mobile SSH client: Termius vs Blink Shell (and the newer agent-focused apps)
- **Termius** — the most popular pick in the Claude Code write-ups. Polished, cross-platform sync (iOS/Android/Mac/Windows/Linux), free tier is enough to start. Downsides: best features (sync, SFTP, port forwarding) are behind a Pro subscription, and it's SSH-only (no native mosh).
- **Blink Shell** — the power-user favorite on Apple. **Native mosh** is the headline: "*Mosh alone is worth the one-time price the first time your train goes into a tunnel.*" Survives sleep/reboot, fast renderer, real shell feel. Cheap annual pricing ($19.99/year). Downside: no push notifications.
- **Secure ShellFish** — uniquely fuses SSH + SFTP into the iOS Files app, has tmux awareness, and added features explicitly for "agentic coding" and "writing instructions for Claude Code."
- **NovaAccess** — newer, built around Tailscale: connects to your tailnet, Tailscale SSH hosts, SFTP file browsing, internal web browsing, all native iOS.
- **Moshi** — newest, purpose-built for AI coding agents: native mosh, tmux pickup, agent notification inbox, Live Activities, Apple Watch approvals, and on-device Whisper voice.

The standing recommendation: **Termius if you want the gentlest start and cross-platform sync; Blink if you live on Apple and value mosh-grade resilience.**

### 4. Persistent sessions and connection drops: tmux + mosh
- **tmux** is the near-universal choice for session persistence (screen and zellij are alternatives but rarely recommended in these write-ups). A useful pattern people use: a no-prefix `PageUp` binding for scrolling on a phone, and `destroy-unattached` so local tabs die but SSH sessions persist.
- **mosh** solves the *connection* problem that tmux doesn't: it uses UDP, roams across Wi-Fi→cellular→Wi-Fi without dropping, survives sleep, and gives instant local echo. The interactivity gain is dramatic: per Winstein & Balakrishnan's mosh paper (mosh.org), over a Sprint EV-DO (3G) link, "Mosh reduced the median keystroke response time from 503 ms to nearly instant (because more than 70% of the keystrokes could be immediately displayed), and reduced the mean keystroke response time from 515 ms to 173 ms." Gotchas: per the official mosh man page (linux.die.net), "By default, mosh uses the ports between 60000 and 61000," and mosh-server "binds to a port between 60000 and 61000 and executes the user's login shell" — so those UDP ports must be reachable (Tailscale handles this on your tailnet). Mosh can be blocked by aggressive public Wi-Fi firewalls, and only does the interactive shell (no port forwarding/tunneling). **mosh + tmux together** is the belt-and-suspenders combo most resilient setups use.

### 5. Your voice problem: what people actually prefer instead of the buggy app
The native Claude Code mobile/app dictation is widely reported as flaky. Anthropic's own GitHub tracker documents it: Issue #44441 describes the cursor jumping "backward ~5 words" and recording "silently" stopping mid-dictation ("No error messages — the failure is silent"), intermittent on "roughly 10-20% of dictation attempts." Issue #29399 argues native phone STT is the wrong foundation: "Native STT (Web Speech API, iOS dictation) times out after ~30 seconds, struggles with technical terms, requires the screen to stay active" — e.g. "deploy to kv-node-7" coming back as "deploy to cave node seven." And critically, **the CLI `/voice` does not work over SSH at all** — per Claude Code's docs it "needs local microphone access" and "does not work in remote environments such as Claude Code on the web or SSH sessions." So for your exact situation, the field has converged on a few middle-grounds:
- **Happy Coder** (free, open-source, MIT; `npm i -g happy`, run `happy` instead of `claude`) — the clear community favorite for a free mobile client (~19,000+ GitHub stars). Crucially, its voice is *not* literal transcription; it's a translation-layer agent. Per Happy's own blog ("Why Voice Coding Makes Sense (Even Though It Sucks)"): "The voice agent isn't smart... It does one thing: it takes your rambling and makes it comprehensible to Claude Code. When you're thinking out loud about code, you backtrack, revise, remember things out of order. Claude Code needs clean, structured requests. The voice agent handles that translation." End-to-end encrypted, runs on your hardware, supports plan mode, custom agents, parallel sessions. A widely-cited user testimonial: "I've tried CodeRemote, YoloCode, Omnara, Claudia, Conductor, Cursor Agents, Kisuke... Terragon Labs. Nothing beats github.com/slopus/happy with Claude Code. Simple, #OpenSource, and free for the win."
- **Omnara** (YC S25) — polished mobile/web/Apple Watch client with a voice agent; runs a daemon with an outbound WebSocket so it needs no SSH/port-forwarding. Per its Launch HN, voice surprised the founders ("we honestly thought coding by talking to a voice agent would be gimmicky... What surprised us is how useful the voice agent ended up being"). **Pricing: free for 10 agent sessions/month, then $20/month for unlimited** (note: this is $20, not the ~$9 you may have seen — ~$9 is closer to Superwhisper's tier). Caveat: no end-to-end encryption yet (conversations stored server-side for sync).
- **Secure ShellFish "Type Sentence" / on-device dictation** — if you want to stay in your terminal, this is the cleanest. Per its App Store changelog, "Type Sentence mode, where you can compose text using autocomplete and other standard iOS editing features. Nothing is sent to the server until you press Return and leave the mode, making it particularly useful for typing natural language while editing in Emacs or Vim, or when writing instructions for Claude Code." A newer build adds on-device dictation (iOS 26) that's "less suited for issuing shell commands."
- **Moshi** — on-device Whisper voice ("running entirely on your device... The audio is discarded immediately after transcription") plus a dedicated conversational "Chat Mode" for AI agents.

The deeper pattern people settle on: **voice the reasoning, type the precise tokens** (file paths, function names, hostnames) — because that's exactly where transcription fails. As one source (claudefa.st) frames it: "Type the file paths and function names, voice the reasoning."

### 6. File browsing / viewing from your phone
Ranked by how well they fit "review markdown plan files and code on the go":
- **Secure ShellFish → iOS Files app (SFTP)** — best low-friction option. Your laptop's filesystem appears in Files alongside iCloud Drive; you can open markdown in iA Writer/Textastic, edit in place, and changes write back over SFTP. Has improved `.ts.net`/Tailscale hostname handling.
- **code-server (VS Code in the browser)** — full editor in a browser tab, exposed only over your tailnet. Tailscale's own docs walk through running it with Caddy + Let's Encrypt `*.ts.net` certs so even clipboard/HTTPS features work. Heaviest but most capable.
- **`tailscale serve`** — the elegant glue: expose any local web service (a markdown preview server, code-server, a dev server on `localhost:3000`) to your tailnet over HTTPS with a stable `https://<machine>.<tailnet>.ts.net` URL, no ports opened to the internet. `opencode serve/web` and VibeTunnel pair naturally with this for browser-based agent review (note: VibeTunnel's browser terminal works on mobile but isn't optimized for small screens — its own users recommend Happy for mobile-first interaction).
- **VS Code Tailscale extension** — browse/edit remote files and open terminals via Tailscale SSH from desktop VS Code.
- **Obsidian/Git sync** — if your plan files live in a vault or repo, syncing them is often simpler than remote-mounting.

### 7. Tailscale's broader toolkit — what's relevant to a solo dev
- **Tailscale SSH** — lets Tailscale handle SSH auth via your tailnet identity/ACLs (no key management; `tailscale up --ssh`). Convenient, and there's a browser-based SSH Console in the admin panel. **Tradeoff worth knowing:** as one Hacker News commenter on the GA announcement warned, if an attacker compromises your Tailscale account "that person will immediately have access to your network AND shell access to all of your boxes, rather than just network access if you use Tailscale with vanilla ssh." Plain SSH-over-Tailscale (keys + Tailscale just for transport) keeps those concerns separate.
- **Tailscale Serve** — expose local services privately on your tailnet (above). Highly relevant.
- **Tailscale Funnel** — exposes a local service to the *public* internet with a real TLS cert. Useful for webhooks/demos; **mostly not what you want** for a private dev laptop, and it puts your machine's hostname on a public Certificate Transparency log. Off by default and double opt-in.
- **Taildrop** — AirDrop-like encrypted file transfer between your own devices across the tailnet; handy for pulling a file off the laptop to your phone.
- **Exit nodes** — route all your phone's internet through your home machine (privacy on public Wi-Fi, or appearing to be at home). Nice-to-have, not core to remote coding.
- **Subnet routers** — expose a whole LAN subnet; overkill for a single laptop.
- **MagicDNS** — gives every device a stable name (`my-mac.tailnet.ts.net`) so you never chase IPs. You'll use this constantly.
- **ACLs** — even solo, worth a tight policy (see security below).

### 8. Keeping the laptop awake
The machine must stay on and awake for any of this to work. On macOS the standard tool is **`caffeinate`**:
- `caffeinate -i` prevents idle sleep; `caffeinate -i -t 7200` for two hours; wrap a command (`caffeinate -i -- ./long_task.sh`) and it auto-releases when the command finishes.
- A clean trick for SSH: put `caffeinate -w $PPID &` in an `~/.ssh/config` `LocalCommand` so the Mac stays awake exactly as long as the SSH process lives.
- **Gotchas:** over SSH there's no display, so skip `-d`; for a truly always-on "server" Mac, configuring power settings / `launchd` (or `pmset`) is more robust than chaining `caffeinate` calls; and a sleeping/lidded laptop can still drop off the network. Many people just leave the machine plugged in with sleep disabled.

### 9. Security and gotchas people wish they'd known
- **Tailscale's deny-by-default ACL flip:** the moment you add your first ACL rule, the default "allow all" is replaced by deny-by-default — this "catches every team the first time." Plan your first ACL carefully or you'll lock yourself out.
- **Identity provider is your real attack surface:** "Tailscale's security is as strong as your identity provider's security." Turn on MFA at your IdP and enable **device approval** so a phished password can't silently add a new device to your tailnet.
- **Key expiry:** per Tailscale Docs, "By default, new domains are set with an expiry period of 180 days," after which nodes must re-authenticate; you can disable expiry for trusted always-on machines, and per Tailscale's "Tagged Nodes No Longer Require Key Renewal" blog, "Starting today, tagged devices will have key expiry disabled by default." Periodically review devices and remove stale ones.
- **ntfy topics are public by default** — anyone who guesses your topic name can read or post. Use a long random topic, or self-host ntfy on your tailnet.
- **code-server / web services over plain HTTP** can be hit by DNS-rebinding; bind to localhost and front with `tailscale serve` (HTTPS) rather than `0.0.0.0`.
- **macOS SSH hardening:** disabling password auth on macOS requires fiddling with PAM; a common pragmatic mitigation is to allow SSH only on the Tailscale interface and use a strong password plus keys.
- **General Tailscale grumbles** from HN: the iOS app has historically been a battery drain when left on all day (one user reported ~20–25% over ten days); and some users are uneasy about a closed-source coordination server having central control (mitigations: Tailnet Lock, or self-hosting Headscale).

## Details

**A concrete recommended build (macOS laptop + iPhone):**
1. Install Tailscale on the laptop and phone, sign in with the same account; enable **MagicDNS**. Confirm `tailscale status` and that you can reach the laptop by name.
2. On the laptop: enable Remote Login (System Settings → General → Sharing), install **tmux** and **mosh** (`brew install tmux mosh`), and keep it awake with `caffeinate -i` (or disable sleep while plugged in).
3. On the phone: install **Termius** (easy start) or **Blink Shell** (mosh, Apple-native). Generate an Ed25519 key in the app, add the public key to `~/.ssh/authorized_keys` on the laptop.
4. Start work inside tmux: `tmux new -s claude`, then run `claude` (or `happy` if you want the mobile UI/voice agent). Detach with `Ctrl-b d`.
5. From the phone, connect over Tailscale (`mosh you@my-mac` or SSH), then `tmux attach -t claude`. You're now driving the exact live session.
6. Add **ntfy** notifications via a Claude Code `Notification`/`Stop` hook so your phone buzzes when Claude needs input or finishes — pairs perfectly with stepping away. Use a random topic or self-host over Tailscale. (Tools like nickknissen/claude-ntfy-hook and coa00/claude-push even add tap-to-Allow/Deny approval buttons on the notification.)
7. For file review, add **Secure ShellFish** to mount the laptop in the iOS Files app; for full-editor review, run **code-server** and expose it with `tailscale serve`.

**On the "should you even do this" question:** several authors (notably esc.sh) make the honest point that this is best reserved for the *one time* you're out and want to quickly review Claude's work, press a key, and put the phone away — *"do you really want to do this? If you're taking a walk, then take the walk. Don't bring Claude with you."* Worth keeping in mind.

## Recommendations
1. **Start with the $0 stack today:** Tailscale + MagicDNS + SSH + tmux + Termius. This alone solves "review/access my laptop and files from my phone." Add **mosh** the first time a dropped connection annoys you (switch to Blink Shell if you want native mosh on iOS).
2. **Fix the voice pain with Happy Coder first** — it's free, open-source, runs on your existing machine, and its voice agent is designed precisely to turn spoken rambling into clean prompts. If you'd rather not add a client and prefer staying in the terminal, use **Secure ShellFish's "Type Sentence"/on-device dictation**, which is purpose-built for composing Claude Code instructions before sending. Adopt the "voice the reasoning, type the paths" habit.
3. **Add ntfy push notifications** so you can leave the agent running and get pinged only when needed — this is the single biggest quality-of-life upgrade reported.
4. **For file/markdown review, use Secure ShellFish → Files.app** as the default; escalate to **code-server behind `tailscale serve`** when you need a real editor.
5. **Try Anthropic's Remote Control** as the zero-setup quick-check path, but keep the tmux stack as your resilient backbone (Remote Control's ~10-minute disconnect timeout and one-session-per-instance limit make it secondary).
6. **Lock down security before you rely on it:** enable IdP MFA + Tailscale device approval, write one careful ACL, disable key expiry only on the always-on laptop, and use random/self-hosted ntfy topics.

**Thresholds that would change the recommendation:** If you're on Claude Max and only ever need quick check-ins, you may not need the SSH stack at all — Remote Control suffices. If you want a *full editor* experience rather than terminal review, jump straight to code-server. If connection drops are constant (subway commute), prioritize mosh/Blink over everything else. If you ever need to share a running service publicly (webhook, client demo), that's the one time Funnel beats Serve.

## Caveats
- **Recency/preview risk:** Claude Code Remote Control launched Feb 25, 2026 as a *research preview* with reported rollout bugs ("Contact your administrator" errors for some Pro/Max users) and is evolving fast; specifics (plan availability, timeouts) may change, and sources already disagree on which plans have access.
- **Vendor-blog bias:** several voice-tool and SSH-client comparisons cited are vendor blogs with promotional intent; their factual specs (pricing, platforms, local-vs-cloud) are consistent across competitors, but treat head-to-head "verdicts" skeptically.
- **Voice A/B evidence is thin:** I did not find rigorous side-by-side Reddit comparisons of Happy Coder vs the official app's voice quality; the strongest signal is community testimony plus Anthropic's own documented native-app bugs (GitHub Issues #44441, #29399).
- **Security model judgment:** Tailscale SSH is convenient but concentrates power in your IdP/tailnet account; for a solo dev this is usually an acceptable tradeoff, but it's a real consideration, not a non-issue.
- **macOS sleep is genuinely finicky** — lid-closed/clamshell behavior, network interface sleep, and power-nap settings vary by model; expect some trial and error to get a laptop reliably reachable 24/7.
