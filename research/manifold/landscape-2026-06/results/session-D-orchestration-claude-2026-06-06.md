# Session D — Orchestration & Agent IDEs: Landscape Scout Report

## TL;DR
- The orchestrator market has bifurcated into **agent-native delivery platforms** (Devin/Windsurf-as-Devin-Desktop, Factory.ai Droids, Cursor Cloud Agents, OpenAI Codex, GitHub Copilot cloud agent, Replit Agent 3/4) that hold intent as an internal prompt + ticket, and **framework runtimes** (LangGraph/LangSmith, Microsoft Agent Framework, CrewAI, AutoGen, OpenHands) that hold no spec at all and expect you to bring one. Almost none of them carry a structured, queryable spec graph — they treat "intent" as Markdown, a Linear ticket, or a chat prompt.
- The only commercial player that has bet the entire product on **structured, internal spec ownership** is AWS Kiro (which deprecates Q Developer); GitHub Spec Kit and Augment Intent represent the **external/agent-agnostic** spec-driven approach. Everyone else — including Devin, Factory, Cursor, Codex, Copilot — treats AGENTS.md, repo Markdown, and PRD links as "good enough."
- The "who holds intent during parallel agent work" gap is real, documented, and currently unowned: spec drift, "agent built the wrong thing," and 8.5x token amplification from context re-ingestion are the dominant 2026 failure modes per the Berkeley MAST taxonomy (41%–86.7% failure rates across 1,642 annotated traces), the Stanford Digital Economy Lab token-cost paper, and named field incidents (Replit/SaaStr July 2025; Amazon/Kiro Cost Explorer outage). This is the precise wedge manifold can occupy if it exposes its SQLite spec graph over MCP.

## Key Findings

1. **Spec ownership is almost universally weak.** Of the 12 platforms inventoried, only **Kiro** (Amazon) and **Augment Intent** treat the spec as a first-class, mutable, system-versioned artifact. **Devin, Factory.ai, Cursor, Codex, Copilot, Windsurf/Devin Desktop, Replit, OpenHands** all consume intent as either (a) AGENTS.md / CLAUDE.md / .factory/droids markdown files in the repo, (b) a Linear/Jira/GitHub Issue, or (c) the latest chat prompt. **LangGraph, CrewAI, AutoGen/MAF** own none of it — they are pure orchestration runtimes.
2. **MCP is now table stakes.** Every commercial platform inventoried supports MCP servers as the integration surface for external tools and (importantly) external context. GitHub Copilot cloud agent ships with MCP enabled by default; Cursor 3 has a 40-tool MCP ceiling; Devin Desktop has built-in MCP plugin slots; Factory uses MCP for its Knowledge Droid; Codex exposes itself as an MCP server. This is the open door for manifold.
3. **The orchestration model is converging on "coordinator + specialist subagents on git worktrees."** Factory's coordinator droid, Cursor's `/multitask` (April 24, 2026) + worktree pattern, Codex's worktree-per-agent command center (Feb 2026), Devin Desktop's Agent Command Center + Kanban, Augment Intent's coordinator/specialist/verifier — all the same shape. Replit Agent 4 advertises "Parallel Agents." This convergence makes the *intent-holding* layer increasingly external to any single agent.
4. **Pricing is moving away from flat seats toward parallel-run / token-metered economics.** Devin Team is $500/seat/mo flat; everything else is now credit/token-metered: Cursor background agents run "Max Mode" ($0.30–$5+ per task), Windsurf Pro $20/mo with daily/weekly quotas, Kiro spec-mode credits cost ~5x vibe-mode credits ($0.20 vs $0.04), Replit Core $25 + usage. **CrewAI Professional** is $25/mo for only 100 executions, $0.50 each over. The "parallel agent run" is the new unit of consumption.
5. **Multi-agent failure is now an active research area, not a hypothetical.** Berkeley's MAST taxonomy (Cemri et al., arXiv:2503.13657) measured **41%–86.7% failure rates across 1,642 annotated execution traces**; "inter-agent misalignment" and "disobey task specification" are the top categories. Stanford Digital Economy Lab (Pei et al., 2026) measured agentic tasks at **~1,000x the token cost of code chat**, with the cost dominated by re-read input tokens.
6. **Q Developer is dead; Kiro is the official replacement.** AWS blocked new Q Developer signups on May 15, 2026, with end-of-support on April 30, 2027. Kiro is explicitly positioned around "specs > prompts" after the December 2025 AWS Cost Explorer outage was traced to an unsupervised Kiro/agentic coding session.
7. **The "spec stays outside the orchestrator" pattern is the de-facto norm.** Linear's own engineering team uses Linear agents to *draft* PRDs and then dispatches Devin/Copilot/Cursor agents from those tickets — the spec lives in Linear, not in the agent. GitHub Spec Kit (~90,000+ stars and 8,000+ forks on GitHub as of May 9, 2026 per MarkTechPost; ~93K stars per a same-day comparison piece) is explicitly built around the premise that the spec must live in `.specify/specs/*.md` in the repo, and agents must be interchangeable behind it.

- Augment quoting field observation: *"AI did not remove complexity. It relocated it. The effort is no longer in writing code. It is in understanding what should be built. Feed an agent a vague requirement and you get working code that does the wrong thing — fast."* — Simon Martinelli, via foojay.io (foojay.io/today/best-practices-for-working-with-ai-agents-subagents-skills-and-mcp/, 2026)

## 3 Implications for manifold

### 1. **The integration target is the MCP slot, not the agent itself.**
Every commercial orchestrator inventoried — Devin Desktop, Factory, Cursor (40-tool ceiling), Codex CLI (which *is* an MCP server), GitHub Copilot cloud agent (MCP-by-default), Windsurf, Replit, OpenHands — exposes MCP servers as the primary integration surface. Cursor's repository settings, Copilot's repo-level MCP config, and Codex's Agents-SDK MCP integration all map to the same shape: **a server that exposes typed queries**. manifold's SQLite spec graph behind an MCP server is therefore not a competing product for these platforms — it is the missing context layer they're built to consume. The integration path with each is identical: ship the MCP server, document the tool schemas, win.

**Threshold to revisit:** if any of Devin/Factory/Cursor/Copilot ships a *native* internal spec graph (not just AGENTS.md, but a structured, queryable, versioned intent store), manifold's MCP play becomes a direct competition with them rather than an integration. Today, only Kiro and Augment Intent have anything in this direction, and both are closed/proprietary.

### 2. **The direct competitors are Kiro and Augment Intent — and the differentiation is M2 (graph) + M7 (MCP-native).**
**Kiro** owns the spec internally but the spec is a flat 3-document set (requirements.md → design.md → tasks.md). It is not a *graph*, it does not expose itself over MCP, and it locks you into a Code-OSS fork. **Augment Intent** has the strongest "living spec" story but it's tied to Augment's proprietary Context Engine and BYOA model. Neither makes the spec a first-class, MCP-queryable, layered graph that other orchestrators can read. That precise gap — layered spec graph + MCP exposure + drift detection — is unoccupied, and Kapoor/Narayanan's reliability argument (reliability improving at half the rate of capability) plus Berkeley's MAST data (41%–86.7% failure rates) give it economic urgency.

**Threshold to revisit:** if Kiro adds an MCP "spec server" that exposes its specs to *other* agents (Cursor, Devin, Codex), or if Augment opens up Intent's living-spec API publicly, the differentiation collapses to UX and indexing depth. Until then, manifold's M2+M7 combination is genuinely open.

### 3. **The "who holds intent during parallel agent work" gap is real, sized, and currently unowned.**
Every platform that runs parallel agents on git worktrees (Cursor `/multitask`, Codex desktop, Factory droids, Cursor 3 Build-in-Parallel, Devin Desktop multi-agent, Replit Agent 4 Parallel Agents) has the same structural problem: the worktrees diverge, the agents have no shared spec, and conflicts surface in code rather than in intent. Carlini's 16-agent compiler experiment showed that without a shared knowledge base, *all 16 agents converged on the same Linux kernel bug*. The MAST taxonomy puts "FM-2.2 Fail to Ask for Clarification" at 6.8% and "FM-2.6 reasoning/action mismatch" at 13.2%. A manifold-shaped "spec held in SQLite, served over MCP, queried by every parallel agent before each tool call" addresses these directly. The future "orchestrator" component in the manifold architecture should not try to *be* the orchestrator (the market has 12+); it should be the *intent-broker* that every orchestrator reads from. The "progress-tracker" telemetry layer matches one-for-one to LangSmith Engine / Devin Agent Command Center / Cursor agents dashboard — also an integration surface, not a competitive one.

**Threshold to revisit:** if the multi-agent failure rate (currently 41%–86.7% per MAST) drops below ~10% through pure model improvements, the "shared intent" wedge weakens significantly. Kapoor/Narayanan's data says this isn't happening: reliability is improving at half the rate of capability on the general agentic benchmark and at one-seventh the rate on the customer-service benchmark.

## Recommendations (staged, with revisit-thresholds)

**Stage 1 — Ship the MCP server (next 4–8 weeks).** Expose the SQLite spec graph as an MCP server with three primary tool calls: `spec.what_next(scope)`, `spec.get_node(id)`, `spec.detect_drift(commit_range)`. Target Cursor, Codex CLI, and GitHub Copilot cloud agent first — they have the largest user bases with the most mature MCP slots and the weakest internal spec stories. Validate by getting a single power-user team to dispatch a Cursor `/multitask` run that queries manifold for "what next" before each subagent kicks off.
- *Threshold to escalate:* one paying customer ships >50 PRs/week through manifold-fed agents. Then move to Stage 2.

**Stage 2 — Build the drift detector (8–16 weeks).** The Berkeley MAST data and the Fiberplane "drift" linter pattern both point to the same primitive: a CI gate that compares spec-node hashes to code-region hashes and flags inconsistencies. This is the M4 axis where nobody — including Kiro — has shipped anything serious. Manifold can own M4 standalone.
- *Threshold to escalate:* if Kiro ships an MCP-exposed spec server, immediately publish a Kiro-compatible spec-import adapter and reposition manifold as "open layer underneath Kiro." If Augment open-sources Intent's living-spec API, build an Intent-importer.

**Stage 3 — Add the orchestrator (16+ weeks, only if Stage 1 traction).** Do *not* compete with Factory/Devin/Cursor on dispatch. Instead, ship a thin "manifold-orchestrator" that consumes the spec graph and the progress-tracker telemetry and produces a *prioritized work queue* that any of the existing orchestrators can pull from. Position: "the work-queue layer for your existing agent fleet."
- *Threshold to kill:* if MAST-style failure rates drop below 10% via better base models — and field incidents (Replit/SaaStr-style) stop appearing — for two consecutive quarters, the "shared intent" thesis weakens and manifold should pivot to pure spec-version-control (a more boring but defensible niche).

**Do not:** try to be an IDE, ship an autonomous agent, or build a model. Every well-funded company in the inventory is doing one or more of those. The unowned territory is the typed, structured, MCP-exposed intent layer underneath them.
