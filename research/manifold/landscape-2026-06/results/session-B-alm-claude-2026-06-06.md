# Session B — Enterprise RE / ALM / Traceability Landscape Scan

## TL;DR
- **Enterprise ALM is rapidly bolting on AI copilots but is NOT becoming agent-native.** Every major vendor (IBM, Siemens, PTC, Jama, Visure, Modern Requirements) shipped an "AI requirements assistant" in 2024–2025, but they remain monolithic, GUI-first, per-seat enterprise products — only Jama Connect explicitly mentions MCP as a "product context layer," and only Modern Requirements/Copilot4DevOps and the agent-IDE wave (AWS Kiro, GitHub Spec Kit, Augment Intent) treat AI agents as first-class clients of the spec graph.
- **The traceability problem manifold targets is real and unsolved at the agent-orchestrator scale.** Enterprise tools solve M1 (canonical truth) + M2 (graph) + M6 (verification) brilliantly for regulated industries but charge ~$50–$80/user/month, require admins, and assume humans (not agents) author requirements — leaving a clear gap for a SQLite/MCP-native, solo-dev tool like manifold.
- **The agent-native frontier is forming around "spec-driven development" (Kiro, GitHub Spec Kit at 108k+ stars, BMAD-METHOD, Augment Intent), not around ALM.** Practitioners on Hacker News and PeerSpot increasingly describe DOORS/Polarion as "cash cows" and migrate to docs-as-code (Sphinx-Needs, Doorstop, StrictDoc) or just markdown-in-git — but those OSS tools still lack drift detection and a navigation/compass layer.

> **M1–M7 interpretation used in this scan** (please correct if wrong, since 00-MASTER-HANDOFF.md was not provided):
> - **M1_truth** = canonical source of truth for intent (one place where the PRD lives)
> - **M2_graph** = layered/linked spec graph structure with typed traceability links (KAOS-like)
> - **M3_history** = history/versioning of intent over time (baselines, diffs of intent)
> - **M4_drift** = drift detection between requirements and implementation (code/test)
> - **M5_compass** = navigation / "what should we work on next" queries against the graph
> - **M6_verdict** = verification/acceptance verdicts against requirements (test-to-req traceability)
> - **M7_agent** = agent-native integration (MCP server, agent-callable API, AI copilot embedded in spec)

## Key Findings

1. **Two distinct moves in 2025–2026.** Legacy enterprise ALM (DOORS, Polarion, Codebeamer, Helix) added generative-AI "Assistants" that improve quality of human-written requirements but treat the AI as a user, not a peer process. A second wave (Jama Connect with MCP-based "product context layer," Modern Requirements Copilot4DevOps, AWS Kiro, GitHub Spec Kit) treats specs as machine-readable contracts AI agents can read/write.
2. **ReqIF is the lowest-common-denominator interop layer** and is supported by virtually every commercial RM tool (DOORS, Polarion, Jama, Visure, Codebeamer, Helix RM, ReqView). Open source Eclipse RMF / ReqIF Studio / formalmind Studio give the OSS world a viewer/editor. Important: ReqIF preserves attributes and hierarchy but is XML/SpecObject-centric — not graph-native and not optimized for agent consumption.
3. **Drift detection is still mostly manual.** None of the commercial enterprise tools surveyed automatically detect when shipped code diverges from a requirement; they detect *link suspect status* (Jama Suspect Triggers, DOORS Suspect Links) when upstream artifacts change. True code↔intent drift detection is being prototyped only in the agent-IDE category (Kiro, Augment Intent, Kinde) and academic tooling.
4. **Pricing tiers create a clear segmentation gap.** Enterprise RM/ALM clusters around ~$50–$80/user/month (Jama, Polarion, Codebeamer, Modern Requirements4DevOps), with DOORS Next on a token model (~$30K+/yr typical for a 10-seat installation per PeerSpot estimates). OSS (Doorstop, Sphinx-Needs, StrictDoc) is free but devops-only. Nothing exists at the "solo-dev / agent-orchestrator" scale manifold targets.
5. **The "PRD as historical fiction" complaint is widely echoed.** Practitioners on Hacker News and PeerSpot explicitly call enterprise RM "too heavy for modern dev workflows" and are reaching for markdown+git; meanwhile AI-spec-driven advocates warn that static specs decay even faster under AI agents — captured in InfoQ's "spec gap," Augment Code's "living specs," and ThoughtWorks Technology Radar Volume 34 (April 2026): *"Even within emerging practices such as spec-driven development, we've noted the risk of reverting to traditional software-engineering antipatterns — most notably, a bias toward heavy up-front specification and big-bang releases."*

### Direct answers to the targeted questions

- **Do practitioners still use ALM, or "just GitHub + markdown" in 2025–2026?** Both — and the split is bimodal. Regulated industries (automotive ISO 26262, medical IEC 62304, aerospace DO-178C) still buy DOORS / Polarion / Jama / Codebeamer because regulators expect baselines and audit trails. Everyone else is migrating to "GitHub + markdown + Sphinx-Needs / Doorstop / StrictDoc" or just running Linear / Jira + Notion PRDs with AI copilots. There is a visible middle layer (manifold's target) that is currently unserved.
- **Complaints about traceability overhead vs AI speed?** Yes, repeatedly. Both from the legacy side ("DOORS is slow, expensive, doesn't fit modern dev workflows") and the AI side (ThoughtWorks Radar Vol 34 "heavy up-front specification" warning, April 2026; InfoQ "spec gap"). The synthesis emerging is: keep traceability, but make it agent-readable and let agents maintain it.
- **Any vendor claiming "AI requirements" or "intent drift detection"?** "AI requirements" — yes, almost all of them (IBM Engineering AI Hub, Polarion Copilot + AI Tender Manager, Jama Connect Advisor, Visure NLP, Codebeamer AI, Copilot4DevOps, requisis_ORCA). "Intent drift detection" at the spec↔code level — **no**, not as a shipped enterprise feature. The closest are Augment Intent's Coordinator/Verifier pattern and Kinde's CI/CD spec-drift checks, both very early.

## 3 Implications for manifold

1. **What enterprise got right that manifold should adopt: typed, baselined, link-rich graphs as the long-term substrate.** DOORS's "Satisfies / Validated-by / Tracked-by" link types, Polarion's LiveDocs work-item graph, Jama's Trace View, and ReqIF's SpecRelation are not bureaucratic noise — they're what makes impact analysis possible 18 months later when an engineer asks "if I change this, what breaks?" manifold should ensure its KAOS-inspired layers include **typed link semantics** (not just edges), **immutable baselines** (M3_history), and **ReqIF import/export** as a cheap interop on-ramp into the enterprise world. The single biggest reason DOORS still exists in 2026 is that regulators trust its baselines and audit trail; manifold's SQLite store can give a small team a credible version of that for free.

2. **What's overkill for the agent era: monolithic UIs, per-seat licensing, and "review cycles" as the unit of work.** Modern Requirements Copilot4DevOps, AWS Kiro, GitHub Spec Kit, and Augment Intent are converging on a different shape: agent-callable, file-based, fast feedback loops measured in minutes — not GUI-heavy "Review Center" workflows measured in days. manifold should not try to be a Jama Connect Advisor; it should be the MCP-callable spec graph that an AI agent reads/writes between commits. Specifically, drop: review/approval gating UIs, multi-license tier modeling, electronic-signature workflows, BIRT-style reporting. Keep: a single canonical store, a drift report runnable in CI, and an MCP endpoint that exposes the "what next" question over the typed graph.

3. **Win the space enterprise ALM and OSS docs-as-code both miss: live drift detection + a compass layer.** Every commercial tool surveyed handles M1, M2, M3, and M6 (truth, graph, history, verdicts) well. Every open-source tool handles M1 and partial M2/M3. **Almost nothing handles M4 (real-time spec↔code drift) and M5 (compass / "what next") in one place.** Kiro / Spec Kit ship `tasks.md` but don't validate against current code; Augment Intent has the right model but is enterprise-priced; Doorstop / Sphinx-Needs have no agent interface. manifold's defensible wedge is the combination: **(a)** a SQLite spec graph that an MCP-aware agent can mutate during a coding session, **(b)** a CI-runnable drift-report that diffs intent vs reality, and **(c)** a compass query that ranks the next unit of intent to work on against current state. That triple is the actual gap in the 2026 market, not "another RM tool."
